## 1. 测试目标与范围

- **测试目标**
  - 验证商城核心业务在多环境下的稳定性、正确性与可回归性。
  - 通过接口自动化 + UI 自动化，构建一套可持续运行的回归体系。
- **范围**
  - 接口自动化：核心业务 API、鉴权、异常、幂等、数据一致性。
  - UI 自动化：关键用户旅程（下单主链路）与高频回归场景。
- **重点业务域**
  - 用户与权限（注册 / 登录 / 找回密码 / 地址）。
  - 商品（分类、检索、详情、库存）。
  - 购物车（增删改查、勾选、合计）。
  - 订单（提交、支付、取消、超时关闭、售后）。
  - 营销（优惠券、满减、会员价）。
  - 支付与回调（成功 / 失败 / 重复回调）。
  - 管理后台（商品、订单、库存、营销配置）。

## 2. 测试策略

- **测试金字塔**
  - 接口自动化为主（约 70%）：稳定、快、易并行。
  - UI 自动化为辅（约 20%）：覆盖关键端到端流程。
  - 冒烟 / 合约检查（约 10%）：每次构建快速反馈。
- **分层设计**
  - 公共层：鉴权封装、请求客户端、数据工厂、断言库、日志。
  - 业务层：按“用户 / 商品 / 订单”等领域封装 API & PageObject。
  - 场景层：下单链路、支付链路、退款链路等多接口、多页面组合。
- **环境策略**
  - dev：日常开发联调。
  - sit：主回归环境。
  - pre：发布前准生产验证。
- **数据策略**
  - 测试账号池（普通用户 / VIP / 商家 / 管理员）。
  - 商品数据分层（有库存 / 无库存 / 上下架）。
  - 用例前置数据通过 API / SQL 建立，执行后自动清理。

## 3. 接口自动化计划

### 3.1 覆盖对象

- 用户中心 API：注册、登录、token 刷新、地址管理。
- 商品中心 API：分类、搜索、详情、库存查询。
- 购物车 API：加购、修改数量、移除、勾选、价格汇总。
- 订单中心 API：创建订单、查询订单、取消订单、确认收货。
- 支付 API：发起支付、支付结果查询、支付回调处理。
- 营销 API：优惠券领取 / 核销、满减计算、活动生效校验。
- 后台 API：商品上下架、库存调整、订单发货。

### 3.2 用例分层

- L1 冒烟（P0）
  - 登录成功、商品查询、加购成功、下单成功、支付成功回调。
- L2 核心业务（P1）
  - 库存扣减一致性、优惠计算准确性、订单状态流转正确性。
- L3 异常与边界（P2）
  - token 过期、参数缺失、重复提交、库存不足、回调重复通知。
- L4 稳定性（P2）
  - 幂等校验、并发下单、重试机制、超时与降级（在接口允许的范围内）。

### 3.3 关键断言

- HTTP 状态码 + 业务码双断言。
- 响应结构 JSON Schema 校验。
- 订单金额：商品总额 - 优惠 + 运费 = 应付金额。
- 订单状态机合法流转（待支付 → 已支付 → 已发货 → 已完成）。
- 数据库一致性（库存、订单、支付流水）。

## 4. UI 自动化计划（Selenium）

### 4.1 覆盖范围

- 登录 / 注册页。
- 首页、搜索页、商品列表页。
- 商品详情页。
- 购物车页。
- 结算页（地址、发票、优惠券）。
- 支付结果页。
- 个人中心（订单列表 / 订单详情）。
- 管理后台（商品管理、订单管理的关键流）。

### 4.2 核心场景（P0）

- 新用户下单主链路：
  - 注册并登录 → 搜索商品 → 详情页加购 → 购物车结算 → 选择地址与优惠 → 提交订单 → 支付 → 订单中心校验。
- 老用户复购链路：
  - 登录 → 购物车勾选多个商品 → 结算 → 支付 → 订单状态与金额校验。
- 下单失败回退：
  - 加购后模拟库存不足 / 价格变动 → 提交订单失败 → 正确提示 & 回退逻辑。

### 4.3 稳定性设计

- Page Object Model + 业务组件封装。
- 元素定位优先 data-testid / 稳定属性，避免纯 XPath。
- 显式等待 + 必要隐式等待 + 失败截图 / 日志。
- 弹窗、iframe、第三方支付跳转等封装专用处理。
- 用例相互独立，可单独执行与重跑。

## 5. 工具与技术栈

- 接口自动化：
  - Python + pytest + requests/httpx + allure-pytest。
- UI 自动化（已按你要求采用 Selenium）：
  - Python + pytest + Selenium + WebDriverManager。
- CI / 报告：
  - Jenkins / GitLab CI / GitHub Actions（按项目实际选择）。
  - Allure Report 或平台自带报告。

## 6. 执行节奏（建议 4 周）

- 第 1 周：
  - 需求 & 接口文档梳理。
  - 自动化框架初始化（当前代码结构已基本完成）。
  - 确定 P0 冒烟集（接口 + UI）。
- 第 2 周：
  - 完成接口 P0 / P1 用例开发与调通。
  - 接入环境配置、数据工厂、基础断言。
- 第 3 周：
  - 完成 UI 主链路（登录 → 下单 → 支付 → 校验）场景。
  - 将接口 & UI 执行接入 CI，形成每日构建。
- 第 4 周：
  - 异常 / 边界场景补齐。
  - 稳定性治理（Flaky 用例分析 & 修复）。
  - 发布前自动化回归基线固化。

## 7. 入口 / 出口标准（DoR / DoD）

- **入口标准（Ready）**
  - 需求与接口说明文档可用，主要字段、错误码、鉴权方式明确。
  - dev / sit 环境服务可用，域名与证书配置完成。
  - 已准备自动化专用账号及基础商品 / 优惠券 / 库存数据。
  - 测试团队完成需求评审，识别关键场景与风险点。

- **出口标准（Done）**
  - P0 / P1 自动化用例执行通过率 ≥ 98% / 95%。
  - 阻塞级、严重级缺陷全部关闭或有明确风险说明。
  - 形成发布前回归报告、风险清单、未覆盖项说明。
  - 自动化脚本已纳入 CI，具备一键执行与结果追踪能力。

## 8. 风险与应对

- 需求变更频繁：
  - 用例与代码按模块解耦，采用标签管理（smoke / regression / module）。
  - 建立变更影响分析机制，仅重跑受影响模块回归集。
- 测试数据污染：
  - 账号池隔离（每条关键 UI 流程使用独立账号）。
  - 前后置脚本对订单、购物车、库存执行清理与回滚。
- UI 不稳定：
  - 统一定位规范（优先 data-testid）。
  - 页面加载与异步请求使用统一等待策略。
  - 失败用例聚类分析，针对性优化。
- 第三方支付不稳定：
  - 使用沙箱环境或 Mock 回调。
  - 测试时重点验证系统对回调结果的处理逻辑（成功、失败、重复）。
